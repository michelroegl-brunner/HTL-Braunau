\chapter{Grundlagen Active Directory}

\section{Was ist Active Directory?}
\textbf{Active Directory (AD)} ist ein von Microsoft entwickelter Verzeichnisdienst, der erstmals mit Windows Server 2000 eingeführt wurde. Ein Verzeichnisdienst ist ein zentraler Speicher für Informationen über Objekte in einem Netzwerk -- Benutzer, Computer, Drucker, Gruppen und viele weitere Ressourcen -- und stellt Mechanismen zur Authentifizierung und Autorisierung bereit.

Active Directory basiert auf dem offenen Protokoll \textbf{LDAP} (Lightweight Directory Access Protocol) und nutzt \textbf{Kerberos} als primäres Authentifizierungsprotokoll. Ergänzend wird \textbf{DNS} (Domain Name System) intensiv genutzt: Ohne funktionierenden DNS-Dienst arbeitet Active Directory nicht zuverlässig.

In der Praxis ist Active Directory der De-facto-Standard für das Identitäts- und Zugriffsmanagement in Windows-basierten Unternehmensumgebungen. Es findet sich in nahezu jeder Organisation, die mehr als ein Dutzend Windows-Rechner betreibt.

\section{Kernkonzepte}

\subsection{Domain}
Eine \textbf{Domain} (Domäne) ist die grundlegende administrative Einheit in Active Directory. Sie bündelt alle Objekte -- Benutzer, Computer, Gruppen -- unter einem gemeinsamen Namensraum, z.\,B.\ \texttt{bannanenimport.local}. Innerhalb einer Domain gelten gemeinsame Sicherheitsrichtlinien.

\subsection{Forest und Tree}
Mehrere Domains können zu einer \textbf{Tree} (Baumstruktur) verbunden werden, wenn sie denselben DNS-Namensraum teilen. Mehrere Trees bilden zusammen einen \textbf{Forest} (Gesamtstruktur), der die oberste Sicherheitsgrenze in Active Directory darstellt. Für Schulungsumgebungen genügt in der Regel eine einzige Domain ohne Forest-Verbund.

\subsection{Organisationseinheiten (OUs)}
\textbf{Organisationseinheiten} (Organizational Units, OUs) sind Container innerhalb einer Domain, mit denen Objekte strukturiert und gruppiert werden. OUs spiegeln häufig die Unternehmensstruktur wider -- etwa nach Abteilungen oder Standorten. Auf OUs können gezielt \textbf{Gruppenrichtlinien} (GPOs) angewendet werden.

\subsection{Benutzer und Gruppen}
\textbf{Benutzerkonten} repräsentieren Personen oder Dienste und enthalten Anmeldeinformationen sowie Profildetails. \textbf{Gruppen} fassen mehrere Benutzer oder Computer zusammen und erleichtern die Verwaltung von Berechtigungen erheblich: Anstatt jedem einzelnen Benutzer Rechte zu erteilen, werden diese der Gruppe zugewiesen.

Es gibt zwei Haupttypen von Gruppen:
\begin{itemize}
  \item \textbf{Sicherheitsgruppen}: für Berechtigungsverwaltung und Gruppenrichtlinien
  \item \textbf{Verteilergruppen}: für E-Mail-Verteilerlisten, keine Berechtigungsverwaltung
\end{itemize}

Gruppen haben außerdem einen \textbf{Gültigkeitsbereich}:
\begin{itemize}
  \item \textbf{Lokal}: nur innerhalb der Domain nutzbar
  \item \textbf{Global}: domainweit, kann in anderen Domains verwendet werden
  \item \textbf{Universal}: forest-weit nutzbar
\end{itemize}

\subsection{Gruppenrichtlinien (GPO)}
\textbf{Group Policy Objects (GPOs)} sind Richtlinienpakete, die auf Benutzer oder Computer angewendet werden. Sie steuern hunderte von Einstellungen: Passwortrichtlinien, Sperrbildschirm-Timeouts, Softwareverteilung, Netzlaufwerk-Mappings, Startskripte und vieles mehr. GPOs werden auf Ebene von Domain, Standort oder OU verknüpft.

\subsection{Domain Controller}
Der \textbf{Domain Controller (DC)} ist der zentrale Server, auf dem Active Directory läuft. Er authentifiziert Benutzer und Computer beim Anmelden, beantwortet LDAP-Anfragen und repliziert die AD-Datenbank bei mehreren DCs. In kleinen Umgebungen genügt ein einziger DC; für Produktivumgebungen empfehlen sich mindestens zwei für Ausfallsicherheit.

\section{Einsatzgebiete}
Active Directory wird in Unternehmen und Bildungseinrichtungen für eine Vielzahl von Aufgaben eingesetzt:

\begin{itemize}
  \item \textbf{Zentrales Benutzermanagement}: Benutzerkonten, Passwörter und Profilinformationen werden an einem Ort verwaltet. Ein Benutzer kann sich mit denselben Anmeldedaten an beliebigen Computern der Domain anmelden.
  \item \textbf{Single Sign-On (SSO)}: Nach einer einmaligen Anmeldung am Betriebssystem haben Benutzer Zugriff auf alle Ressourcen, für die sie berechtigt sind -- ohne erneute Passwortabfrage.
  \item \textbf{Netzwerkressourcen verwalten}: Drucker, Dateifreigaben und Anwendungen werden zentral bereitgestellt und den richtigen Benutzern und Gruppen zugewiesen.
  \item \textbf{Sicherheitsrichtlinien}: GPOs erzwingen einheitliche Sicherheitseinstellungen auf allen Computern der Domain -- z.\,B.\ Mindestpasswortlänge, Bildschirmsperre oder deaktivierter USB-Zugriff.
  \item \textbf{Softwareverteilung}: Anwendungen können automatisch auf Computern oder für Benutzer installiert werden.
  \item \textbf{Audit und Compliance}: Anmeldevorgänge, Zugriffsversuche und administrative Änderungen werden protokolliert.
\end{itemize}



\chapter{Netzwerkplanung mit Proxmox}

\section{Konzept: Isoliertes Labornetz}
Für diese Übung wird ein vollständig isoliertes Netzwerk in Proxmox eingerichtet. Der Vorteil: Die VMs sind vom restlichen Netzwerk und vom Internet getrennt, was einerseits die Sicherheit erhöht und andererseits eine realistische Unternehmensumgebung simuliert, in der der Domain Controller alle Netzwerkdienste (DNS, DHCP) selbst übernimmt.

Das Konzept sieht folgendermaßen aus:

\begin{verbatim}
  +------------------------------- Proxmox Host -------------------------------+
  |                                                                            |
  |   vmbr0 (mit physischer NIC)          vmbr1 (intern, kein Bridge-Port)     |
  |   --> Internet / Schulnetz            --> isoliertes AD-Labornetz          |
  |                                                                            |
  |              +---------------+        +------------------+                 |
  |              |     DC01      |        |    CLIENT01      |                 |
  |              | Windows Srv   |        | Windows 10/11    |                 |
  |              | 192.168.100.1 |        | 192.168.100.x    |                 |
  |              |  (DHCP, DNS)  |        |  (via DHCP)      |                 |
  |              +-------+-------+        +--------+---------+                 |
  |                      |                         |                           |
  |                      +----------vmbr1----------+                           |
  |                                                                            |
  +----------------------------------------------------------------------------+
\end{verbatim}

Der Domain Controller \texttt{DC01} erhält eine feste IP-Adresse und betreibt gleichzeitig den DNS- und DHCP-Dienst für das interne Netz. Der Windows-Client \texttt{CLIENT01} bekommt seine IP-Konfiguration automatisch per DHCP.

\section{Proxmox: Interne Bridge \texttt{vmbr1} anlegen}
Damit das isolierte Netz funktioniert, muss in Proxmox eine zweite Linux-Bridge ohne physischen Netzwerkport angelegt werden:

\begin{enumerate}
  \item Im Proxmox-Webinterface den Knoten (Server) im linken Baum auswählen
  \item Auf \textit{Netzwerk} klicken
  \item Oben auf \textit{Erstellen} $\rightarrow$ \textit{Linux Bridge} klicken
  \item Folgende Einstellungen vornehmen:
  \begin{itemize}
    \item \textbf{Name}: \texttt{vmbr1}
    \item \textbf{Bridge-Ports}: \textit{leer lassen} (kein physischer Port -- rein intern)
    \item \textbf{VLAN aware}: optional, für diese Übung nicht notwendig
    \item Kommentar: \texttt{AD-Labornetz intern}
  \end{itemize}
  \item Auf \textit{Erstellen} klicken, dann oben auf \textit{Änderungen übernehmen}
\end{enumerate}

Die neue Bridge \texttt{vmbr1} erscheint nun in der Netzwerkliste und steht beim Erstellen von VMs als Netzwerkschnittstelle zur Verfügung.

\section{Aufgabe 1: Proxmox-VM für den Domain Controller anlegen}

\subsection*{Ziel}
Eine virtuelle Maschine für den späteren Windows Server / Domain Controller wird in Proxmox erstellt.

\subsection*{Voraussetzungen}
\begin{itemize}
  \item Proxmox VE ist installiert und erreichbar
  \item Die interne Bridge \texttt{vmbr1} wurde wie in Kapitel~2 beschrieben angelegt
  \item Ein Windows-Server-2022-ISO ist auf dem Proxmox-Speicher vorhanden\\
        (ISO-Upload: Speicher \texttt{local} $\rightarrow$ \textit{ISO Images} $\rightarrow$ \textit{Upload})
\end{itemize}

\subsection*{Aufgabe}
\begin{enumerate}
  \item Klicke oben rechts im Proxmox-Webinterface auf \textit{VM erstellen}
  \item Konfiguriere die VM mit folgenden Einstellungen:
  \begin{itemize}
    \item \textbf{Allgemein}: Name \texttt{DC01}, VM-ID nach Wahl (z.\,B.\ \texttt{100})
    \item \textbf{OS}: Windows-Server-2022-ISO auswählen, Typ \texttt{Microsoft Windows}, Version \texttt{11/2022}
    \item \textbf{System}: BIOS \texttt{OVMF (UEFI)}, Maschine \texttt{q35}, TPM 2.0 hinzufügen (für Windows Server 2022 empfohlen)
    \item \textbf{Festplatten}: 60\,GB, Bus/Gerät \texttt{VirtIO SCSI}, Storage nach Verfügbarkeit
    \item \textbf{CPU}: 2 Kerne, Typ \texttt{host}
    \item \textbf{Speicher}: 4096\,MiB (4\,GB)
    \item \textbf{Netzwerk}: Bridge \texttt{vmbr1}, Modell \texttt{VirtIO (paravirtualized)}
  \end{itemize}
  \item Auf \textit{Fertigstellen} klicken -- die VM erscheint im linken Baum
  \item Erstelle vor dem ersten Start einen Snapshot: VM auswählen $\rightarrow$ \textit{Snapshots} $\rightarrow$ \textit{Snapshot aufnehmen}, Name: \texttt{leer-vor-installation}
  \item Starte die VM und öffne die Konsole über \textit{Console}
\end{enumerate}

\subsection*{Hinweis}
VirtIO-Treiber werden von Windows während der Installation möglicherweise nicht erkannt. In diesem Fall muss zusätzlich eine zweite virtuelle CD mit dem \textbf{VirtIO-Treiber-ISO} (kostenlos von \href{https://fedorapeople.org/groups/virt/virtio-win/direct-downloads/stable-virtio/}{fedorapeople.org}) eingebunden werden, damit der Festplatten- und Netzwerktreiber gefunden wird.

\section{Aufgabe 2: Windows Server installieren und Domain Controller promoten}

\subsection*{Ziel}
Windows Server 2022 wird installiert, mit einer statischen IP-Adresse konfiguriert, und der Server wird zum Domain Controller der Domain \texttt{bannanenimport.local} befördert.

\subsection*{Aufgabe -- Teil A: Windows Server installieren}
\begin{enumerate}
  \item Starte die VM \texttt{DC01} und starte die Windows-Server-2022-Installation über die Konsole
  \item Wähle \textbf{Windows Server 2022 Standard (Desktop Experience)} -- die Version mit grafischer Oberfläche
  \item Wähle \textit{Benutzerdefinierte Installation}, wähle das virtuelle Laufwerk aus (VirtIO-Treiber ggf.\ nachladen)
  \item Vergib ein sicheres Administratorpasswort
  \item Nach dem ersten Anmelden: Computernamen auf \texttt{DC01} ändern\\
        (\textit{Systemsteuerung} $\rightarrow$ \textit{System} $\rightarrow$ \textit{Computername ändern}) und neu starten
\end{enumerate}

\subsection*{Aufgabe -- Teil B: Statische IP-Adresse setzen}
\begin{enumerate}
  \item Öffne \textit{Netzwerk- und Freigabecenter} $\rightarrow$ \textit{Adaptereinstellungen}
  \item Rechtsklick auf den Ethernet-Adapter $\rightarrow$ \textit{Eigenschaften} $\rightarrow$ \textit{Internet Protocol Version 4 (TCP/IPv4)}
  \item Setze folgende Werte:
  \begin{itemize}
    \item IP-Adresse: \texttt{192.168.100.1}
    \item Subnetzmaske: \texttt{255.255.255.0}
    \item Standardgateway: \textit{leer lassen} (kein Internet in diesem Netz)
    \item Bevorzugter DNS: \texttt{127.0.0.1} 
  \end{itemize}
\end{enumerate}

\subsection*{Aufgabe -- Teil C: AD DS-Rolle installieren und DC promoten}
\begin{enumerate}
  \item Öffne den \textbf{Server-Manager} $\rightarrow$ \textit{Rollen und Features hinzufügen}
  \item Wähle \textit{Rollenbasierte oder featurebasierte Installation}
  \item Wähle \textbf{Active Directory Domain Services} (AD DS) -- weitere Features werden automatisch vorgeschlagen, bestätigen
  \item Installiere die Rolle und klicke danach auf das Warndreieck im Server-Manager: \textit{Diesen Server zu einem Domänencontroller heraufstufen}
  \item Im Konfigurationsassistenten:
  \begin{itemize}
    \item \textbf{Neue Gesamtstruktur hinzufügen}
    \item Root-Domänenname: \texttt{bannanenimport.local}
    \item Funktionsebene: \textit{Windows Server 2016} oder höher
    \item DNS-Server und Globaler Katalog: aktiviert lassen
    \item DSRM-Passwort (Directory Services Restore Mode) vergeben und notieren
  \end{itemize}
  \item Installation abschließen -- der Server startet automatisch neu
  \item Nach dem Neustart mit \texttt{BANANENIMPORT\textbackslash Administrator} anmelden
\end{enumerate}

\subsection*{Snapshot empfohlen}
Erstelle nach erfolgreichem DC-Promote einen Snapshot in Proxmox: \texttt{dc-promote-fertig}

\section{Aufgabe 3: DHCP-Server einrichten}

\subsection*{Ziel}
Der Domain Controller übernimmt auch den DHCP-Dienst für das interne Netz, sodass Windows-Clients automatisch eine IP-Adresse erhalten.

\subsection*{Aufgabe}
\begin{enumerate}
  \item Öffne den \textbf{Server-Manager} $\rightarrow$ \textit{Rollen und Features hinzufügen}
  \item Wähle die Rolle \textbf{DHCP-Server} und installiere sie
  \item Nach der Installation: Warndreieck im Server-Manager $\rightarrow$ \textit{DHCP-Konfiguration abschließen} $\rightarrow$ Autorisierung mit dem Domain-Administratorkonto bestätigen
  \item Öffne das \textbf{DHCP-Verwaltungstool} (\textit{Start} $\rightarrow$ \textit{DHCP})
  \item Lege einen neuen Bereich (Scope) an:
  \begin{itemize}
    \item Rechtsklick auf \textit{IPv4} $\rightarrow$ \textit{Neuer Bereich}
    \item Name: \texttt{AD-Labornetz}
    \item Startadresse: \texttt{192.168.100.50}
    \item Endadresse: \texttt{192.168.100.150}
    \item Subnetzmaske: \texttt{255.255.255.0}
    \item Ausschlüsse: keine (oder \texttt{192.168.100.1} ist ohnehin außerhalb)
    \item Leasedauer: Standard (8 Tage) oder kürzer für Laborumgebungen (z.\,B.\ 1 Tag)
  \end{itemize}
  \item Konfiguriere die \textbf{Bereichsoptionen}:
  \begin{itemize}
    \item \textbf{003 Router (Gateway)}: \texttt{192.168.100.1}
    \item \textbf{006 DNS-Server}: \texttt{192.168.100.1}
    \item \textbf{015 DNS-Domänenname}: \texttt{bannanenimport.local}
  \end{itemize}
  \item Aktiviere den Bereich (Scope aktivieren)
\end{enumerate}

\section{Aufgabe 6: Logon-Skript pro Abteilung einrichten}

\subsection*{Ziel}
Jeder Benutzer soll beim Anmelden automatisch das Netzlaufwerk seiner Abteilung als Laufwerk \texttt{Z:} eingebunden bekommen. Dies wird über ein PowerShell-Skript realisiert, das per Gruppenrichtlinie (GPO) auf OU-Ebene angewendet wird.

\subsection*{Aufgabe -- Teil A: Logon-Skripte erstellen}

Erstelle für jede Abteilung ein eigenes PowerShell-Skript. Die Skripte werden im SYSVOL-Ordner gespeichert, damit alle Domaincomputer darauf zugreifen können.

Speicherpfad auf dem DC:
\begin{center}
\texttt{C:\textbackslash Windows\textbackslash SYSVOL\textbackslash sysvol\textbackslash bannanenimport.local\textbackslash scripts\textbackslash}
\end{center}

Erstelle dort folgende vier Dateien:

\textbf{Datei \texttt{logon\_IT.ps1}:}
\begin{lstlisting}[language=sh, caption=Logon-Skript IT-Abteilung]
# Netzlaufwerk fuer die IT-Abteilung einbinden
$laufwerk = "Z:"
$pfad     = "\\DC01\IT"

# Vorhandenes Laufwerk Z: trennen (falls vorhanden)
if (Test-Path $laufwerk) {
    net use $laufwerk /delete /yes
}

# Netzlaufwerk verbinden
net use $laufwerk $pfad /persistent:no

Write-Host "Laufwerk $laufwerk wurde mit $pfad verbunden."
\end{lstlisting}

  Erstelle analoge Skripte \texttt{logon\_Buch\allowbreak haltung.ps1}, \texttt{logon\_Vertrieb.ps1} und \texttt{logon\_Personal.ps1} -- jeweils mit dem entsprechenden Freigabepfad (\texttt{\textbackslash\textbackslash DC01\textbackslash Buchhaltung} usw.).

\subsection*{Aufgabe -- Teil B: GPO pro Abteilung erstellen und verknüpfen}
\begin{enumerate}
  \item Öffne die \textbf{Gruppenrichtlinienverwaltung} (\textit{Start} $\rightarrow$ \textit{Windows-Verwaltungstools} $\rightarrow$ \textit{Gruppenrichtlinienverwaltung})
  \item Navigiere zu \textit{bannanenimport.local} $\rightarrow$ \textit{Bananenimport-Fruchticus} $\rightarrow$ \textit{IT}
  \item Rechtsklick auf die OU \texttt{IT} $\rightarrow$ \textit{Gruppenrichtlinienobjekt hier erstellen und verknüpfen}
  \item Name: \texttt{GPO\_Logon\_IT}
  \item Rechtsklick auf \texttt{GPO\_Logon\_IT} $\rightarrow$ \textit{Bearbeiten}
  \item Im Gruppenrichtlinien-Editor navigiere zu:\\
        \textit{Benutzerkonfiguration} $\rightarrow$ \textit{Windows-Einstellungen} $\rightarrow$ \textit{Skripts (Anmelden/Abmelden)}
  \item Doppelklick auf \textit{Anmelden} $\rightarrow$ \textit{Hinzufügen}
  \item Klicke auf \textit{Dateien anzeigen} -- der SYSVOL-Skriptordner öffnet sich
  \item Kopiere \texttt{logon\_IT.ps1} in diesen Ordner (falls nicht bereits dort)
  \item Trage als Skriptname \texttt{logon\_IT.ps1} ein, klicke \textit{OK}
  \item Wiederhole die Schritte für \texttt{Buchhaltung}, \texttt{Vertrieb} und \texttt{Personal}
\end{enumerate}

\subsection*{Hinweis: PowerShell-Ausführungsrichtlinie}
Damit PowerShell-Skripte als Logon-Skript ausgeführt werden dürfen, muss die Ausführungsrichtlinie angepasst werden. Dies kann ebenfalls per GPO erzwungen werden:

\textit{Computerkonfiguration} $\rightarrow$ \textit{Administrative Vorlagen} $\rightarrow$ \textit{Windows-Komponenten} $\rightarrow$ \textit{Windows PowerShell} $\rightarrow$ \textit{Skriptausführung aktivieren} $\rightarrow$ \textit{Allen Skripten erlauben}

\section{Netzwerkdiagramm und Datenfluss beim Login}

Beim Anmelden eines Domainbenutzers laufen folgende Schritte ab:
\begin{enumerate}
  \item Der Client sendet eine \textbf{Kerberos AS-REQ} an den DC (\texttt{192.168.100.1})
  \item Der DC prüft die Anmeldedaten in der AD-Datenbank und stellt ein \textbf{Ticket Granting Ticket (TGT)} aus
  \item Das Betriebssystem verarbeitet die verknüpften \textbf{GPOs} für den Benutzer und den Computer
  \item Das \textbf{Logon-Skript} wird ausgeführt: Laufwerk \texttt{Z:} wird gemappt
  \item Der Benutzer ist angemeldet und hat Zugriff auf seinen Abteilungsordner
\end{enumerate}

\section{Häufige Fehlerquellen}

\subsection*{DNS falsch konfiguriert}
Active Directory ist extrem DNS-abhängig. Der häufigste Fehler ist, dass der Windows-Client den falschen DNS-Server eingetragen hat. Sicherstellen:
\begin{itemize}
  \item DNS auf dem Client (per DHCP oder manuell): \texttt{192.168.100.1}
  \item Test: \texttt{nslookup bannanenimport.local} muss \texttt{192.168.100.1} zurückgeben
  \item Test: \texttt{nslookup DC01.bannanenimport.local} muss \texttt{192.168.100.1} zurückgeben
\end{itemize}

\subsection*{Kerberos-Zeitdifferenz}
Kerberos toleriert maximal \textbf{5 Minuten} Zeitdifferenz zwischen Client und DC. Ist die Uhr eines Clients falsch gestellt, schlägt die Anmeldung mit einem kryptischen Fehler fehl. Lösung:
\begin{itemize}
  \item Sicherstellen, dass DC und Client dieselbe Zeitzone und eine synchronisierte Uhrzeit haben
  \item Im isolierten Lab ohne Internet: Auf dem DC einen internen NTP-Dienst konfigurieren oder Zeit manuell angleichen
\end{itemize}

\subsection*{Logon-Skript wird nicht ausgeführt}
\begin{itemize}
  \item PowerShell-Ausführungsrichtlinie prüfen (muss per GPO auf \textit{RemoteSigned} oder \textit{Unrestricted} gesetzt sein)
  \item Skriptpfad in der GPO exakt prüfen -- Groß-/Kleinschreibung beachten
  \item GPO-Verknüpfung prüfen: Ist die GPO mit der richtigen OU verknüpft?
  \item Mit \texttt{gpresult /r} auf dem Client überprüfen, welche GPOs angewendet wurden
  \item GPO-Verarbeitung erzwingen: \texttt{gpupdate /force}
\end{itemize}

\subsection*{Domain-Beitritt schlägt fehl}
\begin{itemize}
  \item Ping zum DC (\texttt{192.168.100.1}) muss funktionieren
  \item DNS muss korrekt aufgelöst werden (\texttt{nslookup bannanenimport.local})
  \item Administratorpasswort korrekt?
  \item Windows-Firewall auf dem DC: \textit{Datei- und Druckerfreigabe} muss aktiviert sein\\
        (\textit{Systemsteuerung} $\rightarrow$ \textit{Windows Defender Firewall} $\rightarrow$ \textit{App durch Firewall zulassen})
\end{itemize}

\section{Gruppenrichtlinien -- Bonusaufgaben}

Wer die Grundaufgaben fertiggestellt hat, kann folgende zusätzliche GPOs konfigurieren:

\textbf{Passwortrichtlinie für die Domain:}
\begin{itemize}
  \item Gruppenrichtlinienverwaltung $\rightarrow$ \textit{Default Domain Policy} $\rightarrow$ Bearbeiten
  \item \textit{Computerkonfiguration} $\rightarrow$ \textit{Windows-Einstellungen} $\rightarrow$ \textit{Sicherheitseinstellungen} $\rightarrow$ \textit{Kontorichtlinien} $\rightarrow$ \textit{Kennwortrichtlinien}
  \item Empfehlung: Mindestlänge 10 Zeichen, Komplexitätsanforderungen aktiviert, maximales Kennwortalter 90 Tage
\end{itemize}

\textbf{Sperrbildschirm-Timeout:}
\begin{itemize}
  \item Neue GPO auf OU-Ebene oder Domain-Ebene anlegen
  \item \textit{Benutzerkonfiguration} $\rightarrow$ \textit{Administrative Vorlagen} $\rightarrow$ \textit{Systemsteuerung} $\rightarrow$ \textit{Anpassung} $\rightarrow$ \textit{Kennwortgeschützten Bildschirmschoner aktivieren} und \textit{Wartezeit für den Bildschirmschoner festlegen} (z.\,B.\ 10 Minuten)
\end{itemize}

\textbf{Softwareverteilung (MSI-Pakete):}
\begin{itemize}
  \item \textit{Computerkonfiguration} $\rightarrow$ \textit{Software-Einstellungen} $\rightarrow$ \textit{Softwareinstallation}
  \item MSI-Datei auf einer Netzwerkfreigabe ablegen und per GPO zuweisen -- wird beim nächsten Computerstart automatisch installiert
\end{itemize}

\textbf{USB-Zugriff sperren (z.\,B.\ für Buchhaltung):}
\begin{itemize}
  \item GPO auf OU \texttt{Buchhaltung} anwenden
  \item \textit{Computerkonfiguration} $\rightarrow$ \textit{Administrative Vorlagen} $\rightarrow$ \textit{System} $\rightarrow$ \textit{Wechseldatenträgerzugriff} $\rightarrow$ \textit{Alle Wechseldatenträgerklassen: Jeglichen Zugriff verweigern}
\end{itemize}
